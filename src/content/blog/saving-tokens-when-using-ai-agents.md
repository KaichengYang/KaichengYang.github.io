## TL;DR

- Use the `opusplan` model so Opus only handles planning and Sonnet does the rest.
- Turn on `CLAUDE_CODE_SIMPLE_SYSTEM_PROMPT=1` to shave ~12k tokens off every turn with Opus 4.7.
- Be deliberate about caching: stay in the conversation when it's still warm, `/clear` when it's not.
- Lower thinking effort by default and only crank it up for planning or tricky issues.
- Trim your context: turn off unused MCP servers, uninstall skills you don't use, keep `claude.md` short.
- RTK and Caveman exist, but the savings are smaller than you'd think --- file reads dominate.

## Background

If your Claude Code subscription tokens also burn too fast, here are some tricks to try before paying Anthropic more money.

## Some Claude Code-specific tricks

### Use the `opusplan` model

The `opusplan` model uses Opus in plan mode, then automatically switches to Sonnet for execution and everything else.

This is my default for Claude Code these days.
Sonnet is very capable at executing the plan generated by Opus, so it works really well in practice.
That said, planning tends to be much more token-heavy, so this model might not save you as many tokens as you'd hope.
But I still highly recommend it.

The `opusplan` model used to be in the model selection menu, but it got removed somehow.
To invoke it, type the following directly in Claude Code:

```
/model opusplan
```

### Simple system prompt

Claude Code lets you use a shorter system prompt for Opus 4.7 by setting an environment variable:

```
export CLAUDE_CODE_SIMPLE_SYSTEM_PROMPT=1
```

You'll need to set this in your bash/zsh environment, or just drop it in your `.bashrc` or `.zshrc`.

Here's the description from the [official documentation](https://code.claude.com/docs/en/env-vars):

> Set to 1 to use a shorter system prompt and abbreviated tool descriptions on Opus 4.7. Has no effect on other models. The full tool set, hooks, MCP servers, and CLAUDE.md discovery remain enabled.

Here are the numbers from my own setup (Opus 4.7, 200k context window, since I don't use the 1m one):

| Context category | Simple system prompt off | Simple system prompt on |
|-----------------|---------------------------|--------------------------|
| System prompt | 9k tokens (4.5%) | 2.8k tokens (1.4%) |
| System tools | 10.4k tokens (5.2%) | 5k tokens (2.5%) |

That's a 12k (6%) difference, which is pretty significant given that this gets sent to the server every turn.

I have been turning this feature on for a while and haven't noticed any drop in quality.
My understanding is that state-of-the-art models have been trained specifically on agentic tasks and don't require extensive system prompts to understand their roles and environments.
This is also why a simple agent harness like the Pi agent can still work really well even with minimal tools and system prompts.

## Universal tricks 

The following tricks apply to other AI agents as well.

### Cache prompts and clear your conversation strategically

Almost all model providers charge a much lower rate for cached prompts (e.g., 10%).
The exact mechanism of prompt caching is complicated and beyond the scope of this post.
A simplified version: if you continue your conversation, it will likely get cached.
But the cache will expire if you clear your conversation or switch models in the middle.
The cache may also expire automatically after a while (e.g., 1 hour).

When your conversation has not been cached for whatever reason, it might be more token-efficient to start from a clean slate.
For instance, you can use `/clear` in Claude Code to do this.

Based on these characteristics, my suggestions are:

1. Are you trying to continue the task you've been working on for a while? If so, try to stay in the same conversation with the same model. Otherwise, start from scratch.
2. Has it been over an hour since your last message? If so, the cache has likely expired, and it's more token-efficient to start fresh. If not, continue your conversation.

### Use lower thinking efforts

Thinking models let you choose the thinking effort.
As you'd expect, higher effort gives stronger performance but costs more tokens.

But my experience is that I don't need high thinking effort most of the time --- only when making plans or dealing with tricky issues.
Lowering the thinking effort also reduces latency.
So I have been leaning on this strategy more and more in my daily routine.

The `opusplan` model in Claude Code is one way to implement this.
I also found that gpt-5.5 is a pretty capable model without thinking.
So my default reasoning level in Codex is low (the lowest in Codex), but I set it to high for planning mode.
When I use gpt-5.5 in Pi agent, I turn off thinking completely (see my [blog post](https://www.kaichengyang.me/blog/my-pi-agent-configurations/) for the configuration).

### Manage your context

I mentioned the simple system prompt trick in Claude Code.
There are other things that live in your default prompt and get sent to the server every turn, so you want to clean those up too.

In particular:

- **Turn off unused MCP servers.** MCP servers tend to inject tons of information into your context window. The least you can do is disable the ones you don't need. I also suggest finding CLI replacements when possible (for example, the `gh` command is much better than the GitHub MCP).
- **Uninstall unused skills.** Skill descriptions get added to your context window every time. They tend to be shorter than MCP server descriptions, but they become a problem when you have many of them.
- **Trim your `claude.md` and `agent.md` files.** These files get included in your context window every time. Keep them short and well-organized.

### RTK

As described on its GitHub page, [RTK](https://github.com/rtk-ai/rtk) is a CLI proxy that reduces LLM token consumption by 60--90% on common dev commands.

The idea is that most CLI tools are built for humans, but AI won't need the same level of verbose output.
RTK is designed to shorten the output of bash commands when AI agents interact with your local environment.

Here's an example showing how it works. When I run `git status`, the output looks like this:

```
$ git status

On branch main
Your branch is up to date with 'origin/main'.

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   path/to/your/file.md
```

With RTK, you wrap the bash command:

```
$ rtk git status

* main...origin/main
~ Modified: 1 files
   path/to/your/file.md
```

As you can see, the output is significantly shorter.

The concept is brilliant and it works really well.
I have RTK set up for all my AI agents.

That said, the actual difference it makes to total token consumption is much smaller than you'd expect.
The reason is that the most token-consuming part is reading file contents, not parsing the output of bash commands.

### Caveman

[Caveman](https://github.com/juliusbrussee/caveman) is a plugin that makes the agent respond more succinctly to save tokens.

I haven't used it myself.
A lot of my tasks involve writing, so I worry it might affect the writing quality.
I'm also skeptical about its effectiveness at reducing total token consumption, for the same reason I mentioned in the RTK section.

And I came across this [video](https://www.youtube.com/watch?v=wijoYNiZq3M) which compares the plugin with a simple "be brief" prompt in `agent.md`.
Caveman doesn't seem to provide a consistent advantage.

So my takeaway is that if you want shorter responses from the agent, it's probably better to define the response style yourself in `agent.md`.
